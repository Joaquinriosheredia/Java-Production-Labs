# ADR-0001 — Multi-Stage Build with Layered JARs and Non-Root User

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 09_docker_optimization

---

## Context

A naive Docker build produces a ~500MB image with:
- Full JDK at runtime (unnecessary — only JRE needed)
- Single fat JAR layer: every code change rebuilds 150MB of dependencies
- App runs as root (security risk in shared environments)
- Build context includes `.git/`, `target/`, IDE files (slow transfer)

---

## Decision

Apply 4 optimizations:
1. **Multi-stage build**: JDK for compile, JRE for runtime
2. **Layered JAR** (Spring Boot 3.x): separate layers for dependencies, spring-boot-loader, snapshot-deps, and application code
3. **Non-root user**: `adduser appuser` in runtime stage
4. **Container-aware JVM flags**: `-XX:+UseContainerSupport`, `-XX:MaxRAMPercentage=75.0`

---

## Image Size Comparison

| Image | Base | Size | Root? |
|-------|------|------|-------|
| Naive (JDK, fat JAR) | eclipse-temurin:21-jdk | ~520MB | Yes |
| Optimized (JRE, layers) | eclipse-temurin:21-jre-alpine | ~195MB | No |
| Buildpack (Spring Boot) | Paketo | ~280MB | No |

**62% size reduction.**

---

## Layer Cache Benefit

Without layering:
- Code change → rebuild 150MB fat JAR layer → push 150MB

With layering:
- Code change → rebuild ~2MB application layer → push ~2MB
- Dependencies layer: cached (unchanged until pom.xml changes)

---

## Consequences

- Build time: +30s (multi-stage) but CI cache compensates
- `.dockerignore` required to exclude `target/`, `.git/`, IDE files
- `JarLauncher` class changed between Spring Boot 2.x and 3.x (`launch.JarLauncher`)
- Memory limit must be set in Kubernetes PodSpec for `-XX:MaxRAMPercentage` to work
