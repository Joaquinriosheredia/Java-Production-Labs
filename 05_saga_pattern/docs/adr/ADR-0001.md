# ADR-0001 — Choreography-based Saga over Orchestration

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 05_saga_pattern

---

## Context

A purchase order requires coordinating multiple services: payment and inventory.
Both must succeed, or both must compensate (rollback).
Traditional distributed transactions (2PC) are not suitable for microservices: they block resources
and require all participants to be available simultaneously.

Saga pattern solves this by breaking the transaction into a sequence of local transactions,
each publishing an event that triggers the next step. On failure, compensating transactions undo
prior steps.

---

## Decision

Use **Choreography-based Saga** (event-driven, no central coordinator).

Each service:
1. Listens to its trigger event via Kafka
2. Executes its local transaction
3. Publishes a success or failure event

Compensation: failure events trigger reverse actions (refund, cancel).

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| Orchestration-based Saga (e.g. Conductor) | Central orchestrator is a single point of failure; tight coupling |
| 2-Phase Commit (2PC) | Blocks resources, requires XA transactions, doesn't scale |
| Best-effort (fire and forget) | No consistency guarantee — silent failures |
| Eventual consistency with reconciliation jobs | Works but adds significant complexity for auditing |

---

## Trade-offs

**Pros (Choreography):**
- Loose coupling: services don't know about each other
- Each service independently deployable
- Easy to add new steps (just subscribe to an event)
- No orchestrator SPOF

**Cons (Choreography):**
- Harder to trace a saga end-to-end (need correlation ID + distributed tracing)
- Cyclic dependencies possible if events not designed carefully
- Business logic spread across multiple services (harder to see the "big picture")
- Testing requires spinning up all services and Kafka

---

## Consequences

- Correlation: `orderId` is used as Kafka message key — all events for an order are on the same partition
- Observability: `SagaStatus` field on `PurchaseOrder` provides single-query state tracking
- Idempotency: each Kafka listener must handle duplicate messages (Kafka delivers at-least-once)
- Compensating transactions must be idempotent too

---

## Implementation Note: Lab Simplification

**This lab collapses all saga participants into a single Spring component for self-containment.**

`OrderSagaOrchestrator` hosts `@KafkaListener` methods for three logical services, each identified
by its `groupId`:

| `groupId` | Logical service | In production |
|---|---|---|
| `payment-service` | Processes payment | Separate deployed microservice |
| `inventory-service` | Reserves stock | Separate deployed microservice |
| `order-service-complete` / `order-service-cancel` | Finalises order | Separate deployed microservice |

Collapsing them into one class is an **intentional simplification** to avoid requiring three
separately deployed services in a learning context. The choreography semantics (event-driven,
no central coordinator, each participant reacts independently) are faithfully demonstrated —
only the deployment boundary is collapsed.

In a real production system:
- Each `groupId` would be a distinct microservice with its own repository, database, and deployment pipeline.
- Failure isolation would be real: a crash in the inventory service would not affect the payment service.
- Distributed tracing (OpenTelemetry + Jaeger) would be required to follow a saga end-to-end across services.
