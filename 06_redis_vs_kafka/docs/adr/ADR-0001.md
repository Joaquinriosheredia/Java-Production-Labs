# ADR-0001 — Redis Pub/Sub for Ephemeral Events, Kafka for Durable Streams

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 06_redis_vs_kafka

---

## Context

Two common messaging patterns: ephemeral notifications (cache invalidation, live updates)
and durable event streams (audit log, event sourcing, async processing).
Both Redis Pub/Sub and Kafka are used for messaging. Understanding their trade-offs
determines which is appropriate for a given use case.

---

## Decision

Use **Redis Pub/Sub** for:
- Cache invalidation signals (fire-and-forget, no persistence needed)
- Real-time notifications (chat, live dashboards)
- Low-latency event fan-out

Use **Kafka** for:
- Durable event log (events must survive consumer downtime)
- Event sourcing
- Multiple independent consumers at different offsets
- Replay capability

---

## Trade-offs

| Dimension | Redis Pub/Sub | Kafka |
|-----------|--------------|-------|
| Latency | Sub-millisecond | 1–50ms |
| Durability | Ephemeral — lost if no subscriber online | Durable — configurable retention |
| Replay | No | Yes (seek to offset) |
| Consumer groups | No built-in (use Streams for that) | Yes — independent consumers |
| Ordering | No partition model | Per-partition ordering |
| Throughput | High (in-memory) | Very high (log-structured) |
| Complexity | Low | Higher (Zookeeper/KRaft, partitions, consumer groups) |

---

## Consequences

- Redis: producers don't block on slow consumers (fire-and-forget), but slow consumers miss messages
- Kafka: consumers can lag behind; backpressure is visible as consumer group lag
- For cache invalidation: Redis Pub/Sub is correct (missing a message just means slightly stale cache)
- For orders: Kafka is correct (missing an event = lost money)
