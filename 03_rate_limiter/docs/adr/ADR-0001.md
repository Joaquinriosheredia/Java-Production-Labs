# ADR-0001 — Use Redis Token Bucket over In-Memory Rate Limiting

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 03_rate_limiter

---

## Context

Rate limiting a distributed service requires a shared state store. In-memory rate limiters don't work
when multiple app instances run behind a load balancer: each instance has its own bucket, so the
effective limit becomes `N × limit` where N is the instance count.

---

## Decision

Use **Bucket4j** with a **Redis backend** (Lettuce driver) for a distributed token bucket algorithm.

Each API client gets a bucket in Redis keyed by their API key or IP. The bucket state (token count, last refill time) is stored atomically in Redis, ensuring consistent rate limiting across all app instances.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| In-memory (ConcurrentHashMap) | Not distributed — breaks with multiple app instances |
| Redis INCR + TTL (sliding window) | Doesn't support burst capacity; less precise |
| Nginx rate limiting | Infrastructure-level only, no per-user customization from app |
| Spring's built-in rate limiting | Doesn't exist — no native support |

---

## Trade-offs

**Pros:**
- Distributed: consistent across all app instances
- Token bucket supports burst (e.g. 20 burst, 10/s steady)
- Standard `Retry-After` and `X-RateLimit-*` headers
- Bucket4j handles Redis CAS atomically

**Cons:**
- Redis is a required dependency (single point of failure if not HA)
- Each request adds a Redis round-trip (~0.5ms local, ~2ms prod)
- "Fail open vs fail closed" decision when Redis is unavailable

---

## Consequences

- Redis must be HA (Redis Sentinel or Redis Cluster) in production
- Implement circuit breaker around Redis calls to handle outage gracefully
- Rate limit config (capacity, refill rate) should be externalized to avoid redeployment for tuning
