# ADR-0001 — Use Virtual Threads over Platform Threads for I/O-bound workloads

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 01_virtual_threads

---

## Context

Traditional Java web servers handle concurrent requests using a platform thread pool (typically Tomcat's executor with a fixed size). Each thread consumes OS resources (~1MB stack by default) and blocks during I/O operations (DB queries, HTTP calls, file reads). This limits concurrency to the pool size.

For I/O-heavy workloads, the bottleneck is not CPU but thread availability: 200 concurrent users × 200ms avg I/O = the system needs 40 threads busy at all times just for I/O wait.

Java 21 introduces **Virtual Threads** (Project Loom) as a stable feature. A virtual thread is a lightweight JVM-managed thread that unmounts from its carrier OS thread when blocking on I/O, allowing the OS thread to serve other virtual threads.

---

## Decision

Configure Spring Boot 3.2+ to use `Executors.newVirtualThreadPerTaskExecutor()` as Tomcat's executor, enabling virtual threads for every HTTP request.

```java
@Bean
public TomcatProtocolHandlerCustomizer<?> virtualThreadsCustomizer() {
    return protocolHandler ->
        protocolHandler.setExecutor(Executors.newVirtualThreadPerTaskExecutor());
}
```

Or via `application.yml`:
```yaml
spring.threads.virtual.enabled: true
```

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| Increase platform thread pool size | Linear memory growth, OS scheduler pressure beyond ~500 threads |
| Reactive programming (WebFlux) | High cognitive overhead, callback hell, incompatible with blocking libraries |
| Async/CompletableFuture everywhere | Infects the codebase, poor stack traces, manual executor management |
| Virtual threads via Project Reactor | Reactor adds complexity that Loom makes unnecessary for I/O cases |

---

## Trade-offs

**Pros:**
- Write synchronous, blocking code — no reactive complexity
- Near-unlimited concurrency for I/O-bound work (millions of virtual threads possible)
- Drop-in replacement — no code changes needed for most Spring Boot apps
- ThreadLocal and stack traces work normally

**Cons:**
- Virtual threads are not beneficial for CPU-bound workloads (still need platform threads for computation)
- Pinning risk: `synchronized` blocks pin the virtual thread to its carrier thread; use `ReentrantLock` instead
- JVM profilers may need updates to handle virtual thread cardinality

---

## Consequences

- Tomcat's thread pool replaced by virtual thread executor
- All Spring MVC request handlers run on virtual threads
- Pinning analysis required before adopting third-party blocking code (JDBC, legacy libraries)
- JVM flags `-Djdk.tracePinnedThreads=full` added to monitoring configuration for pinning detection
