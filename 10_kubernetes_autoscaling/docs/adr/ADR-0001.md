# ADR-0001 — HPA on Custom Prometheus Metrics over CPU-Only Scaling

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 10_kubernetes_autoscaling

---

## Context

Kubernetes HPA by default scales on CPU utilization. For I/O-bound Java services (using virtual threads),
CPU utilization stays low even under high request load. CPU-only HPA would never scale up the service,
causing request queuing and degraded latency.

---

## Decision

Use **HPA v2** with custom metrics from Prometheus:
- Primary metric: `lab_active_requests_gauge` (per-pod active request count)
- Secondary: CPU utilization (fallback)

When `avg(lab_active_requests_gauge) > 10`, add pods.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| CPU-only HPA | Doesn't trigger for I/O-bound virtual-thread services |
| VPA (Vertical Pod Autoscaler) | Requires pod restart to resize — not suitable for live traffic |
| KEDA (event-driven) | Good for queue-based scaling; overkill for HTTP request count |
| Manual scaling | Not production-grade |

---

## Prerequisite: prometheus-adapter

Custom metrics require deploying `prometheus-adapter` to bridge Prometheus metrics to the Kubernetes Custom Metrics API:
```yaml
# prometheus-adapter rule
- seriesQuery: 'lab_active_requests_gauge{namespace!="",pod!=""}'
  resources:
    overrides:
      namespace: {resource: "namespace"}
      pod: {resource: "pod"}
  name:
    matches: ^lab_active_requests_gauge$
    as: lab_active_requests_gauge
  metricsQuery: avg(<<.Series>>{<<.LabelMatchers>>}) by (<<.GroupBy>>)
```

---

## Trade-offs

**Pros:**
- Scales on what actually matters: request concurrency
- Works for I/O-bound, CPU-bound, and memory-bound workloads
- `stabilizationWindowSeconds` prevents flapping

**Cons:**
- Requires prometheus-adapter deployment and configuration
- Custom metric must be available before HPA can function (chicken-and-egg on first deploy)
- Scale-down latency (5min window) means over-provisioning for transient spikes

---

## Consequences

- Readiness probe must be correctly configured — HPA doesn't send traffic to unready pods
- `preStop` sleep (5s) gives load balancer time to deregister before pod terminates
- `terminationGracePeriodSeconds` (40s) > app graceful shutdown (30s) — prevents SIGKILL mid-request
