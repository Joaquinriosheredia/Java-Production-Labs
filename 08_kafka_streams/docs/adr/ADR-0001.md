# ADR-0001 — Kafka Streams over Apache Flink for In-Process Stream Processing

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 08_kafka_streams

---

## Context

Real-time aggregation of order events: count orders per user in 60-second windows.
This requires stateful stream processing with windowing.

---

## Decision

Use **Kafka Streams** (embedded in the Spring Boot application) instead of a separate Flink cluster.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| Apache Flink | Requires separate cluster, complex deployment, overkill for this scale |
| Spark Streaming | Micro-batch (not true streaming), heavy JVM footprint |
| Custom Kafka consumer with manual state | Reinventing windowing — bug-prone, no fault tolerance |
| Redis HSCAN + expiry | No true streaming, no replay, no backpressure |

---

## Trade-offs

**Pros:**
- Runs in-process: no separate cluster to manage
- Fault-tolerant state: uses Kafka changelog topics for state store recovery
- Exactly-once processing available (with `processing.guarantee=exactly_once_v2`)
- Standard Kafka ecosystem

**Cons:**
- State store (RocksDB) adds memory and I/O overhead
- Cannot reuse state across Kafka Streams instances without KIP-311
- For very large state (TB), Flink is more appropriate

---

## Consequences

- State is stored in local RocksDB backed by Kafka changelog topics
- Restarting the app replays from the changelog — no state loss
- Throughput scales by adding partitions to input topic + increasing parallelism
