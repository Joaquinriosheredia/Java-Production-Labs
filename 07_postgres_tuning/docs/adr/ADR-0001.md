# ADR-0001 — Partial Indexes and Query Rewriting for Hot-Path Optimization

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 07_postgres_tuning

---

## Context

A poller queries `SELECT * FROM events WHERE status = 'PENDING'` every second.
The table has 1M rows. PENDING events are ~5% of the total (50K rows).
A full-table sequential scan for 5% selectivity is wasteful and degrades as the table grows.

---

## Decision

1. **Partial Index**: `CREATE INDEX idx_events_pending ON events(occurred_at) WHERE status = 'PENDING'`
   - Index contains only PENDING rows (~5% of table)
   - Index size: 5% of a full index = dramatically smaller
   - INSERT/UPDATE cost reduced (only PENDING rows affect the index)

2. **Query rewrite**: Ensure the query matches the partial index condition exactly

---

## EXPLAIN ANALYZE — Before (seq scan)

```
Seq Scan on events (cost=0.00..32847.00 rows=50000 width=150)
  Filter: ((status)::text = 'PENDING'::text)
  Rows Removed by Filter: 950000
Planning Time: 0.5ms
Execution Time: 285ms
```

## EXPLAIN ANALYZE — After (index scan)

```
Index Scan using idx_events_pending on events (cost=0.42..4821.00 rows=50000 width=150)
Planning Time: 0.3ms
Execution Time: 12ms
```

**23× improvement.**

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| Full index on status | Large index (100% of rows), INSERT cost for all rows |
| Partition by status | Complex, PENDING is transient — partition management overhead |
| Materialize PENDING view | Refresh cost; still needs index |
| Application-level cache | Adds staleness, invalidation complexity |

---

## Trade-offs

**Pros:**
- 23× query improvement on test dataset
- Index stays small as PENDING rows are processed (rows removed from index)
- No application changes required

**Cons:**
- Partial index only helps when the query matches the condition exactly
- `WHERE status IN ('PENDING', 'RETRY')` would not use this index
- Index becomes stale if status distribution changes dramatically

---

## Consequences

- Monitor index size with: `SELECT pg_size_pretty(pg_relation_size('idx_events_pending'))`
- Add `pg_stat_statements` to identify next hot queries
- `VACUUM ANALYZE events` after bulk loads to update statistics
