# ADR-0001 — Transactional Outbox Pattern for At-Least-Once Delivery

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 04_outbox_kafka

---

## Context

When a service must both persist a business entity (to PostgreSQL) AND notify other services (via Kafka), the naive dual-write approach creates a consistency problem:

```
// DANGEROUS: dual-write without atomicity
orderRepository.save(order);          // Step 1: DB commit
kafkaTemplate.send("orders", event);  // Step 2: Kafka publish
// If Kafka fails here: order exists in DB but event never delivered
// If app crashes after step 1: same problem
```

This leads to silent data loss: the order is confirmed to the user but downstream services never know it happened.

---

## Decision

Use the **Transactional Outbox Pattern**:
1. Persist the business entity AND an outbox event record in the **same DB transaction**
2. A background poller reads `PENDING` outbox events and publishes them to Kafka
3. On successful publish, mark events as `PUBLISHED`

```
BEGIN TRANSACTION
  INSERT INTO orders (...)
  INSERT INTO outbox_events (aggregate_id, event_type, payload, status='PENDING')
COMMIT

-- Background poller (every 1s)
SELECT * FROM outbox_events WHERE status = 'PENDING'
kafka.send(event)
UPDATE outbox_events SET status = 'PUBLISHED'
```

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| Dual-write (save + send) | Not atomic — silent failures possible |
| Kafka Transactions (exactly-once) | Requires Kafka consumer coordination, complex setup |
| Debezium CDC | Better for large scale; adds infra complexity (Kafka Connect) |
| Saga with compensation | Correct but overkill for simple event notification |

---

## Trade-offs

**Pros:**
- Atomicity guaranteed by DB transaction
- Kafka downtime doesn't lose data — events wait in outbox
- Simple: standard Spring + JPA
- Observable: can monitor PENDING count as a lag metric

**Cons:**
- At-least-once delivery — consumers must be idempotent
- Polling adds latency (up to `poll-interval-ms`, default 1s)
- High-volume scenarios may need multiple pollers (partitioned by aggregate type)
- Outbox table grows — need periodic cleanup of PUBLISHED events

---

## Consequences

- Consumers process `OrderCreated` events idempotently (check if already processed by event ID)
- Outbox PENDING count is a critical metric — spike means delivery lag
- Add `poll-interval-ms` to external config for tuning without redeployment
