# ADR-0001 — Use Resilience4j over Hystrix for Circuit Breaking

**Date**: 2024-01-15
**Status**: Accepted
**Lab**: 02_resilience

---

## Context

Distributed systems make calls to external services that can fail, slow down, or become unavailable.
Without protection mechanisms, a failing downstream service causes:
- Request pile-up (thread pool exhaustion)
- Cascading failures across the call graph
- Degraded UX with no graceful fallback

We need Circuit Breaker, Retry, and Bulkhead patterns to protect the application.

---

## Decision

Use **Resilience4j 2.x** with Spring Boot AOP integration.

Configuration via `application.yml`, annotations on service methods:
```java
@Bulkhead(name = "externalService", fallbackMethod = "bulkheadFallback")
@CircuitBreaker(name = "externalService", fallbackMethod = "circuitBreakerFallback")
@Retry(name = "externalService")
public String callExternalService(String requestId) { ... }
```

---

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| Netflix Hystrix | Deprecated (maintenance mode since 2018), incompatible with Java 21 |
| Spring Retry alone | No circuit breaker — endless retries on sustained failure |
| Manual try/catch | No metrics, no state machine, not configurable at runtime |
| Polly (.NET) | Wrong ecosystem |

---

## Trade-offs

**Pros:**
- Lightweight, no Netflix OSS dependency
- Works with virtual threads (no thread pool model)
- Micrometer metrics out of the box (`resilience4j_circuitbreaker_*`)
- Actuator health endpoint shows circuit state
- Composable: annotations stack (order matters: Bulkhead → CB → Retry)

**Cons:**
- AOP-based: annotations only work on Spring-managed beans called from outside the bean
- Complex interaction between Retry and CircuitBreaker: Retry fires before CB counts failures
- Configuration requires understanding state machine transitions

---

## Consequences

- All external calls decorated with `@CircuitBreaker` + `@Retry` + `@Bulkhead`
- Grafana dashboard monitors: `resilience4j_circuitbreaker_state`, `resilience4j_retry_calls_total`
- Fallback methods must have same signature + exception parameter
- Sliding window size (10) and wait duration (10s) tuned per service SLA
